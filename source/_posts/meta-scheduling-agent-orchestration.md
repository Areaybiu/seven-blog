你有没有这种体验：同时开了 Claude Code 写前端、Codex 审查代码、Cursor 调试 Bug，三个窗口来回切，结果 Claude Code 改了文件 A，Codex 还在审查旧版本的文件 A，最后代码冲突，一团糟。

**问题出在哪？每个 Agent 都是独立的，没有"人"来协调它们。**

<!-- more -->

## 从单 Agent 到多 Agent，再到元调度

先理清三个层次：

```
第一层：单 Agent
- 一个 AI 干所有活
- 例子：ChatGPT、Claude

第二层：多 Agent
- 多个 AI 分工协作
- 例子：CrewAI、AutoGen

第三层：元调度
- 管理多个 Agent 的"总管"
- 例子：Ruflo、Claude Squad、Omnigent
```

**元调度 = 调度 Agent 的 Agent**

| 层次 | 角色 | 比喻 |
|------|------|------|
| 单 Agent | 干活的 | 员工 |
| 多 Agent | 团队协作 | 项目组 |
| 元调度 | 管理团队 | 经理 |

**经理不干活，但管理干活的人。**

## 为什么需要元调度？

### 问题一：工具碎片化

```
你现在可能同时在用：
- Claude Code（写代码）
- Codex（审查代码）
- Cursor（IDE 集成）
- Aider（终端编码）
- Gemini（分析文档）

问题：
- 每个工具都是独立的
- 没法统一管理
- 没法协同工作
```

### 问题二：上下文丢失

```
场景：你在 Claude Code 里写了一半的代码，想切到 Cursor 继续

问题：
- Claude Code 的上下文不会自动同步到 Cursor
- 你得手动复制粘贴，或者重新描述需求
- 效率低到离谱
```

### 问题三：权限控制缺失

```
场景：你让 AI 帮你改代码，结果它把生产环境的数据库删了

问题：
- 没有权限控制
- AI 可以做任何事
- 出了问题追悔莫及
```

### 问题四：跨设备协作困难

```
场景：你在公司电脑上用 Claude Code 写了一半，回家想继续

问题：
- 上下文在公司电脑上
- 回家后得重新开始
- 没法跨设备同步
```

## 三大元调度框架

目前主流的元调度框架有三个：

| 框架 | 核心思想 | GitHub Stars | 开发者 |
|------|----------|--------------|--------|
| **Ruflo** | Agent 集群 | 56k+ | rUv |
| **Claude Squad** | 并行管理 | 6.8k | SMTG AI |
| **Omnigent** | 策略治理 | 2.9k | Databricks |

下面逐个详解。

---

## 框架一：Ruflo —— 最火的 Agent 集群

### 一句话

**让 Claude Code 变成 100+ Agent 的集群。**

### 核心特点

Ruflo 是目前最火的元调度框架，它把多个 Claude Code 实例组织成一个**自学习、自组织的集群**。

```
┌─────────────────────────────────────────────────────────┐
│                    Ruflo 元调度                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │              协调层（大脑）                       │   │
│  │  - 任务分配                                      │   │
│  │  - 负载均衡                                      │   │
│  │  - 冲突解决                                      │   │
│  └─────────────────────────────────────────────────┘   │
│                           │                             │
│         ┌─────────────────┼─────────────────┐          │
│         ▼                 ▼                 ▼          │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐   │
│  │  Agent 集群  │  │  Agent 集群  │  │  Agent 集群  │   │
│  │  (机器 A)   │  │  (机器 B)   │  │  (机器 C)   │   │
│  ├─────────────┤  ├─────────────┤  ├─────────────┤   │
│  │ 编码 Agent  │  │ 测试 Agent  │  │ 部署 Agent  │   │
│  │ 审查 Agent  │  │ 搜索 Agent  │  │ 文档 Agent  │   │
│  │ 优化 Agent  │  │ 分析 Agent  │  │ 监控 Agent  │   │
│  └─────────────┘  └─────────────┘  └─────────────┘   │
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │              共享层（记忆）                       │   │
│  │  - 任务历史                                      │   │
│  │  - 学习经验                                      │   │
│  │  - 最佳实践                                      │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 核心能力

| 能力 | 说明 |
|------|------|
| **自组织** | Agent 自动分配任务，不需要人工干预 |
| **自学习** | 从每次任务中学习，不断改进 |
| **跨机器** | 不同机器上的 Agent 可以协作 |
| **联邦通信** | Agent 之间通信，但不泄露敏感数据 |
| **记忆系统** | 记住之前的工作，避免重复 |

### 安装和使用

```bash
# 安装
npm install -g ruflo

# 初始化
ruflo init

# 启动集群
ruflo start

# 查看状态
ruflo status
```

### 实际场景

**场景：开发一个完整的 Web 应用**

```bash
# 1. 启动 Ruflo 集群
ruflo start

# 2. 分配任务
ruflo task add "开发前端页面" --agent frontend
ruflo task add "开发后端 API" --agent backend
ruflo task add "编写测试用例" --agent testing
ruflo task add "部署到服务器" --agent devops

# 3. 监控进度
ruflo monitor
```

Ruflo 会自动：
- 把任务分配给合适的 Agent
- 协调 Agent 之间的依赖关系
- 解决冲突（比如两个 Agent 改了同一个文件）
- 保存中间结果，支持断点续传

### 适合场景

- ✅ 大型项目开发：需要多个 Agent 协作
- ✅ 多人团队协作：不同人用不同的 Agent
- ✅ 需要跨机器部署：分布式 Agent 集群
- ✅ 需要自学习：Agent 越用越聪明

## 框架二：Claude Squad —— 并行 Agent 管理

### 一句话

**同时跑多个 Agent，互不干扰。**

### 核心特点

Claude Squad 是一个**终端应用**，让你在同一个窗口里管理多个 Claude Code、Codex、Aider 等 Agent，每个 Agent 有独立的工作区。

```
┌─────────────────────────────────────────────────────────┐
│                  Claude Squad                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │              终端界面                             │   │
│  │  ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐       │   │
│  │  │ Tab 1 │ │ Tab 2 │ │ Tab 3 │ │ Tab 4 │       │   │
│  │  └───────┘ └───────┘ └───────┘ └───────┘       │   │
│  └─────────────────────────────────────────────────┘   │
│                           │                             │
│         ┌─────────────────┼─────────────────┐          │
│         ▼                 ▼                 ▼          │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐   │
│  │  工作区 1   │  │  工作区 2   │  │  工作区 3   │   │
│  ├─────────────┤  ├─────────────┤  ├─────────────┤   │
│  │ Agent: Claude│  │ Agent: Codex│  │ Agent: Aider│   │
│  │ 任务: 写前端 │  │ 任务: 写后端 │  │ 任务: 写测试 │   │
│  │ 分支: feat/ │  │ 分支: feat/ │  │ 分支: test/ │   │
│  │   frontend  │  │   backend   │  │   suite     │   │
│  └─────────────┘  └─────────────┘  └─────────────┘   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 核心能力

| 能力 | 说明 |
|------|------|
| **独立工作区** | 每个 Agent 有自己的 git 分支和目录 |
| **并行执行** | 多个 Agent 同时工作 |
| **互不干扰** | 一个 Agent 的错误不影响其他 |
| **快速切换** | 像切换浏览器标签页一样切换 Agent |
| **冲突检测** | 自动检测 git 冲突，提醒你处理 |

### 安装和使用

```bash
# 安装
brew install claude-squad

# 或者用 Go 安装
go install github.com/smtg-ai/claude-squad@latest

# 启动
claude-squad
```

### 实际操作

```bash
# 启动后，你会看到一个终端界面

# 按 'a' 添加新的 Agent
# 选择 Agent 类型（Claude Code / Codex / Aider）
# 输入任务描述

# 按 Tab 切换不同的 Agent
# 按 's' 查看所有 Agent 的状态
# 按 'm' 合并某个 Agent 的代码
```

### 适合场景

- ✅ 并行开发多个功能：前端、后端、测试同时进行
- ✅ 快速原型开发：多个方案同时尝试
- ✅ 需要隔离环境：每个功能一个独立分支
- ✅ 个人开发者：一个人管理多个任务

## 框架三：Omnigent —— 策略治理最强

### 一句话

**管理不同 Agent，统一策略控制。**

### 核心特点

Omnigent 是 Databricks 开源的**元编排框架**，最大的特点是**策略引擎**——你可以定义谁能做什么、不能做什么。

```
┌─────────────────────────────────────────────────────────┐
│                    Omnigent 元调度                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │              策略引擎（权限控制）                  │   │
│  │  ALLOW: 放行                                     │   │
│  │  DENY: 拒绝                                      │   │
│  │  ASK: 等人工审批                                  │   │
│  └─────────────────────────────────────────────────┘   │
│                           │                             │
│         ┌─────────────────┼─────────────────┐          │
│         ▼                 ▼                 ▼          │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐   │
│  │ Claude Code │  │   Codex     │  │   Cursor    │   │
│  │             │  │             │  │             │   │
│  │ 策略: 全权限 │  │ 策略: 只读  │  │ 策略: 需审批 │   │
│  └─────────────┘  └─────────────┘  └─────────────┘   │
│         │                 │                 │          │
│         └─────────────────┼─────────────────┘          │
│                           ▼                             │
│  ┌─────────────────────────────────────────────────┐   │
│  │              云沙箱（隔离执行）                    │   │
│  │  - Modal                                        │   │
│  │  - Daytona                                      │   │
│  │  - E2B                                          │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 核心能力

| 能力 | 说明 |
|------|------|
| **策略引擎** | 三级策略：ALLOW（放行）、DENY（拒绝）、ASK（审批） |
| **跨设备同步** | 电脑开始 → 手机继续 |
| **云沙箱** | 代码在云端隔离执行 |
| **花费控制** | 限制 API 调用次数和费用 |
| **Agent 定义** | 用 YAML 定义 Agent，简单易懂 |

### Agent 定义示例

```yaml
# my-agent.yaml
name: code-reviewer
description: 代码审查 Agent
prompt: |
  你是一位严格的代码审查员。
  检查代码的质量、安全性、最佳实践。
  给出具体的改进建议。

tools:
  - read_file
  - search_code
  
permissions:
  # 只允许读取，不允许修改
  - action: read
    target: "*"
    effect: ALLOW
  - action: write
    target: "*"
    effect: DENY
  - action: execute
    target: "*"
    effect: ASK  # 执行命令需要人工审批
```

### 安装和使用

```bash
# 安装
pip install omnigent

# 初始化
omnigent init

# 定义 Agent
omnigent agent create --from-file my-agent.yaml

# 启动
omnigent start

# 运行任务
omnigent run "审查这个 PR 的代码" --agent code-reviewer
```

### 适合场景

- ✅ 企业环境：需要严格的权限控制
- ✅ 合规要求：需要审计日志
- ✅ 跨设备协作：电脑和手机之间切换
- ✅ 成本控制：限制 API 调用费用

## 如何选择？

### 选 Ruflo 如果你：

- 在做大型项目，需要多个 Agent 协作
- 需要跨机器部署 Agent
- 想要 Agent 越用越聪明（自学习）
- 团队有多人，需要协作

### 选 Claude Squad 如果你：

- 是个人开发者
- 想并行开发多个功能
- 不想折腾复杂配置
- 需要快速上手

### 选 Omnigent 如果你：

- 在企业环境工作
- 需要严格的权限控制
- 需要审计日志
- 需要跨设备同步

## 实战建议

### 1. 从 Claude Squad 开始

如果你是第一次用元调度，建议从 Claude Squad 开始。它最简单，几分钟就能上手。

### 2. 明确分工

每个 Agent 只做一件事。不要让一个 Agent 同时写前端和后端。

### 3. 用好 git 分支

每个 Agent 一个分支，完成后合并。这样即使出了问题，也不会影响其他 Agent。

### 4. 设置权限

特别是用 Omnigent 时，一定要设置好权限。不要让 AI 有删除生产数据库的权限。

### 5. 监控和日志

元调度系统比单 Agent 复杂得多，必须有完善的监控和日志。

## 总结

| 框架 | 一句话总结 | 核心优势 | 核心劣势 |
|------|-----------|----------|----------|
| **Ruflo** | Agent 集群，越用越聪明 | 自组织、自学习、跨机器 | 复杂度高 |
| **Claude Squad** | 并行管理，简单高效 | 简单易用、快速上手 | 功能有限 |
| **Omnigent** | 策略治理，安全可控 | 权限控制、审计日志 | 较新，生态不完善 |

元调度是 AI Agent 发展的必然趋势。当单 Agent 的能力达到天花板，多 Agent 协作就成了刚需。而当多 Agent 系统变得复杂，元调度就成了必需品。

选择合适的框架，让你的 AI Agent 们学会"团队协作"，才是正确的做法。
